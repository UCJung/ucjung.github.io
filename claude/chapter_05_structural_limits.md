# 심화편 도입. AI Agent가 가진 구조적 한계

---

## 왜 심화편이 필요한가?

기초편에서 설치와 도구 활용법을 익혔다면, 이제 중요한 질문이 남습니다.

> **"AI Agent는 왜 가끔 엉뚱한 방향으로 가는가?"**

답은 간단합니다. **AI Agent는 사람이 일할 때 생기는 문제를 고스란히 갖고 있습니다.** 사람 동료에게 발생하는 오해, 독단, 삽질의 패턴이 AI Agent에게도 똑같이 나타납니다. 다만 AI Agent는 이 문제를 **더 빠른 속도로, 더 큰 규모로** 만들어냅니다.

심화편은 이 구조적 한계를 이해하고, 실무에서 통제하는 방법을 다룹니다.

---

## 0. AI Agent의 4가지 구조적 한계

### 한계 ① — 지시가 없으면 자기 판단으로 실행한다

AI Agent는 별도의 지시가 없으면 **자신이 알고 있는 범위 내에서 지시를 해석하고 실행**합니다.

사람도 마찬가지입니다. 신입사원에게 "이거 처리해줘"라고만 하면, 자기가 아는 방식대로 합니다. 결과물을 보고 "이게 아닌데"라고 하면 이미 시간은 지난 뒤입니다.

AI Agent는 이 문제가 더 심합니다. 사람이라면 모르는 부분에서 멈추거나 물어보지만, AI Agent는 **모르는 부분을 추론으로 채워서 끝까지 실행**합니다. 자신이 추론한 부분이 맞는지 틀린지 구분하지 않고, 마치 확신하는 것처럼 진행합니다.

```
[사람 동료의 경우]
  "이 부분은 잘 모르겠는데, 어떻게 할까요?" → 물어봄

[AI Agent의 경우]
  "이 부분은 명시되지 않았으니... JWT가 적절하겠군" → 실행해버림
  (사용자는 이 판단이 있었는지조차 모름)
```

**위험한 이유**: AI Agent가 어떤 부분을 추론으로 판단했는지 사용자가 알 수 없습니다. 결과물만 보면 모든 것이 지시대로 된 것처럼 보이지만, 내부적으로 AI Agent가 임의로 결정한 부분이 섞여 있습니다.

---

### 한계 ② — 대화가 길어지면 지침보다 대화 흐름을 우선한다

AI Agent는 CLAUDE.md에 지침이 정의되어 있어도, **대화가 지속되면 대화 내용을 다음 지시를 실행할 때 우선**합니다.

이것도 사람과 같습니다. 프로젝트 초기에 합의한 원칙이 있어도, 매일 회의에서 나오는 긴급 요청에 밀려 원칙이 흐려지는 것과 동일합니다.

AI Agent의 컨텍스트 윈도우는 200K 토큰입니다. 대화가 쌓이면서 지침 문서는 전체 컨텍스트의 극히 일부가 되고, 최근 대화 내용이 압도적인 비중을 차지하게 됩니다. AI Agent는 가장 최근에 주고받은 맥락을 중심으로 다음 행동을 결정합니다.

```
[세션 초반]
  CLAUDE.md 지침: "모든 변경은 승인 후 적용"
  → 지침을 잘 따름

[세션 중반 — 대화 50회 이후]
  사용자: "이것도 수정해줘"
  사용자: "저것도 바꿔줘"
  → 대화 흐름상 "바로 실행" 패턴이 형성됨
  → 승인 요청 없이 바로 수정하기 시작
```

**위험한 이유**: 사용자는 지침이 여전히 지켜지고 있다고 생각하지만, AI Agent는 이미 대화 흐름에 의해 지침을 사실상 무시하고 있을 수 있습니다.

---

### 한계 ③ — 문제가 풀리지 않으면 악순환에 빠진다

특정 문제가 발생했을 때 해결이 되지 않으면, AI Agent는 **① 자기 판단으로 실행**하고 **② 대화 흐름에 매몰**되면서 점점 깊은 수렁에 빠집니다.

이 과정은 다음과 같이 진행됩니다:

```
[악순환 사이클]

  문제 발생
    ↓
  AI Agent가 자기 판단으로 해결 시도
    ↓
  단편적인 해결 — 근본 원인이 아닌 증상만 처리
    ↓
  파생 문제 발생
    ↓
  파생 문제를 또 자기 판단으로 해결 시도
    ↓
  또 다른 파생 문제...
    ↓
  대화가 길어지면서 원래 목표와 지침을 잃어버림
    ↓
  결과: 코드 전체가 의도와 다른 방향으로 변형
```

사람 개발자도 디버깅에 빠지면 비슷한 일이 벌어집니다. "이 에러 잡으려고 고쳤더니 저쪽이 깨지고, 저쪽 고치니까 또 다른 곳이..." 하지만 사람은 어느 시점에서 "이건 근본적으로 접근이 잘못된 것 같다"고 판단하고 멈출 수 있습니다. AI Agent는 이 판단이 약합니다. **멈추지 않고 계속 파고들며**, 한 번 잡은 방향을 쉽게 바꾸지 않습니다.

**위험한 이유**: 10분간의 악순환이 만들어낸 코드 변경은, 되돌리는 데 30분 이상 걸릴 수 있습니다. 변경이 여러 파일에 걸쳐 있으면 git revert로도 깔끔하게 돌리기 어렵습니다.

---

### 한계 ④ — 모호한 부분은 추론으로 채운다 (그리고 말하지 않는다)

한계 ①의 연장이지만 별도로 강조해야 할 만큼 중요합니다.

AI Agent는 지시에서 모호하거나 빠진 부분을 **추론으로 판단**합니다. 그리고 **추론했다는 사실 자체를 사용자에게 알리지 않습니다.**

```
[사용자 지시]
> 결제 API를 만들어줘. 토스페이먼츠 연동으로.

[AI Agent가 추론으로 채운 부분 — 사용자는 모름]
  · 결제 상태 관리: 직접 상태 머신 구현 (vs. 토스 웹훅 의존)  ← 추론
  · 에러 처리: 3회 재시도 후 실패 처리 (vs. 즉시 실패)       ← 추론
  · 부분 환불: 미구현 (범위에 없다고 판단)                    ← 추론
  · 동시 요청: 낙관적 락 사용 (vs. 비관적 락)                 ← 추론
  · 로깅: console.log 사용 (vs. 구조화된 로거)                ← 추론
```

결과물만 보면 잘 동작하는 것 같습니다. 하지만 위의 5가지 판단 중 하나라도 의도와 다르면, 나중에 발견했을 때 수정 비용이 큽니다. 특히 아키텍처 수준의 판단(상태 관리 방식, 동시성 처리)은 나중에 바꾸려면 전체 구조를 뜯어고쳐야 할 수 있습니다.

**위험한 이유**: 사용자가 "지시를 내렸으니 결과가 지시대로일 것"이라고 믿는 순간, 검증 없이 다음 단계로 넘어갑니다. 추론으로 채워진 부분이 문제를 일으키는 것은 보통 3~4단계 후입니다.

---

## 4가지 한계의 공통 원인

이 한계들은 모두 하나의 근본 원인에서 비롯됩니다:

> **AI Agent는 "모르겠습니다"라고 말하는 것보다 "그럴듯하게 실행하는 것"에 최적화되어 있다.**

사람이라면 모르는 것을 모른다고 하고, 확신이 없으면 확인을 요청합니다. AI Agent는 모르는 것도 아는 것처럼, 확신이 없는 것도 확신이 있는 것처럼 행동합니다.

따라서 심화편에서 다루는 모든 대응책은 결국 하나의 원칙으로 수렴합니다:

```
┌──────────────────────────────────────────────────────────┐
│                                                          │
│   AI Agent가 스스로 판단하는 영역을 최소화하고,            │
│   판단이 필요한 부분은 사람이 명시적으로 결정한다.          │
│                                                          │
│   → 명확한 지시 (심화 1장)                                │
│   → 결과물을 파일로 보존하여 검증 (심화 1장)               │
│   → 구조화된 지침으로 판단 기준 제공 (심화 2장)             │
│                                                          │
└──────────────────────────────────────────────────────────┘
```
