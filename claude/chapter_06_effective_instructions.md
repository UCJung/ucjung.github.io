> 이 장은 도입부에서 다룬 4가지 구조적 한계를 실무에서 통제하는 구체적 방법을 다룹니다.

## 1.1 AI Agent에 명확한 지시가 필요한 이유

### AI Agent는 "추론"하지만 "추측"도 한다

AI Agent는 지시가 모호하면 **스스로 빈 곳을 채워서** 작업을 진행합니다.  
사람이라면 "이거 어떻게 할까요?"라고 되묻겠지만, AI Agent는 가장 그럴듯한 방향으로 판단하고 실행합니다.

이것이 강점이자 위험입니다.

```
[모호한 지시]
> 로그인 기능 만들어줘

[AI Agent의 해석 — 사용자는 모름]
  • 프레임워크? → React + Express로 추정
  • 인증 방식? → JWT로 추정
  • DB? → SQLite로 추정
  • 비밀번호 암호화? → bcrypt로 추정
  • 소셜 로그인? → 없다고 추정

→ 결과: 동작하지만, 의도와 다른 기술 스택으로 구현됨
```

```
[명확한 지시]
> NestJS + Passport.js로 로그인 API를 만들어줘.
> - 인증: JWT 기반, refresh token 포함
> - DB: PostgreSQL + Prisma
> - 비밀번호: argon2 해싱
> - 소셜 로그인은 이번 단계에서 제외

→ 결과: 의도대로 구현됨
```

### "잘 만들어줘"는 지시가 아니다

AI Agent에게 "잘 해줘", "적절하게 해줘", "알아서 해줘"는 **실질적 정보가 0**입니다.

| 모호한 지시 | 문제 | 명확한 지시 |
|------------|------|------------|
| "API 만들어줘" | 스택, 구조, 인증 모두 추정 | "NestJS REST API, JWT 인증, Prisma ORM" |
| "테스트 추가해줘" | 범위, 프레임워크 불명확 | "Jest로 UserService의 단위 테스트 작성" |
| "이거 고쳐줘" | 무엇이 문제인지 불명확 | "POST /users에서 중복 이메일 시 500 대신 409 반환하도록 수정" |
| "성능 개선해줘" | 어디를, 얼마나? | "getUsers 쿼리가 3초 걸림. N+1 문제 확인 후 1초 이내로 개선" |
| "리팩토링해줘" | 기준 없음 | "auth 모듈을 controller/service/repository 3계층으로 분리" |

### 모호한 지시가 만드는 실제 문제

**1. 되돌리기 비용**

AI Agent가 잘못된 방향으로 수십 개 파일을 수정한 뒤 "이게 아닌데"라고 하면, 되돌리는 데 원래 작업보다 더 많은 시간이 듭니다. 명확한 지시 한 줄이 되돌리기 30분을 절약합니다.

**2. 컨텍스트 낭비**

AI Agent가 방향을 잡기 위해 불필요한 파일을 탐색하면 컨텍스트 윈도우가 소모됩니다. 200K 토큰 중 절반이 "방향 잡기"에 쓰이면 정작 구현할 공간이 부족해집니다.

**3. 누적 오차**

첫 번째 지시가 모호하면 이후 모든 작업이 잘못된 전제 위에 쌓입니다. 3단계 뒤에야 문제를 발견하면, 1단계부터 다시 해야 합니다.

### 좋은 지시의 7가지 요소

| 요소 | 설명 | 예시 |
|------|------|------|
| **Why** | 이것을 하려는 목적과 의도 | "회원 이탈률을 줄이기 위해 프로필 수정 UX를 개선하려 한다" |
| **What** | 무엇을 할 것인가 | "사용자 프로필 수정 API 구현" |
| **How** | 어떤 기술/방법으로 | "NestJS, Prisma, class-validator" |
| **Scope** | 범위와 경계 | "이메일, 닉네임만. 비밀번호 변경은 제외" |
| **Reference** | 참고할 자료·조사 대상 | "Prisma 공식 문서(prisma.io/docs) 참고", "src/auth/README.md 참조" |
| **Constraint** | 제약 조건 | "기존 테스트가 깨지지 않아야 함" |
| **Verify** | 검증 방법 | "npm test 통과 + 수동으로 PUT /users/:id 테스트" |

**Why가 중요한 이유**

AI Agent는 목적을 모르면 기술적으로 "동작하는 코드"를 만들지만, 비즈니스 의도에 맞는 코드를 만들지는 못합니다. 목적과 의도를 알면 AI Agent가 애매한 판단을 할 때 올바른 방향으로 추론할 확률이 높아집니다.

| Why가 없을 때 | Why가 있을 때 |
|--------------|-------------|
| "프로필 수정 API 만들어줘" → 기계적으로 CRUD 구현 | "이탈률 줄이려고 UX 개선 중" → 입력 유효성 검사를 UX 친화적으로, 에러 메시지를 사용자 관점으로 작성 |
| 판단 기준 없이 기본값으로 구현 | 목적에 맞는 판단 (응답 속도 우선, 상세 에러 메시지 등) |

**Reference가 중요한 이유**

AI Agent는 학습 데이터 기반으로 답변하므로, 최신 버전이나 프로젝트 고유 규칙과 다를 수 있습니다.  
참고 자료를 명시하면 추측 대신 근거에 기반한 작업이 됩니다.

| 상황 | Reference 지시 예시 |
|------|-------------------|
| 최신 API 사용법 | "https://docs.nestjs.com/security/authentication 페이지를 참고해서 구현해줘" |
| 사내 문서 기반 | "docs/api-convention.md 규칙을 따라서 작성해줘" |
| 기존 코드 패턴 참조 | "src/auth/auth.service.ts의 패턴을 참고해서 동일한 구조로 만들어줘" |
| 외부 라이브러리 | "https://github.com/oraios/serena 의 README를 읽고 설정해줘" |
| 디자인 시스템 | "Figma 링크의 UI를 참고해서 컴포넌트 구현해줘" |

> 💡 Claude Code는 웹 검색과 파일 읽기가 가능하므로,  
> URL을 주면 직접 조회하고, 파일 경로를 주면 직접 읽습니다.  
> "알아서 찾아봐"보다 **"여기를 참고해"** 가 훨씬 정확한 결과를 만듭니다.

모든 지시에 7가지 전부가 필요한 건 아닙니다.  
하지만 **Why**와 **What**과 **Scope**는 항상, **Reference**는 최신 기술이나 사내 규칙이 관련될 때 포함하는 습관이 중요합니다.

### 실전 패턴: 단계별 지시

복잡한 작업은 한 번에 지시하지 말고 단계별로 나눕니다.

```
# ❌ 한 번에 모든 것
> 사용자 관리 시스템을 만들어줘. 회원가입, 로그인, 프로필 수정, 
> 권한 관리, 이메일 인증까지 전부 포함해줘.

# ✅ 단계별 진행
> [1단계] 먼저 사용자 관리 시스템의 DB 스키마를 설계해줘.
>   - User, Role, Permission 테이블
>   - Prisma 스키마로 작성
>   → 리뷰 후 다음 단계 진행

> [2단계] User CRUD API를 NestJS로 구현해줘.
>   - POST /users (회원가입)
>   - GET /users/:id (프로필 조회)
>   - class-validator로 입력 검증
>   → 테스트 통과 확인 후 다음 단계

> [3단계] JWT 인증을 추가해줘.
>   - Passport.js + JWT Strategy
>   - access token 15분, refresh token 7일
>   → 기존 테스트가 깨지지 않는지 확인
```

### Plan Mode 활용

Claude Code의 Plan Mode를 사용하면 실행 전에 계획을 먼저 세우고 승인 후 진행할 수 있습니다.

```
# Plan Mode 진입
> /plan

# 또는 지시에 포함
> 인증 시스템을 OAuth2로 마이그레이션하는 계획을 먼저 세워줘.
> 실행하지 말고 계획만 보여줘. 승인 후 진행할게.
```

> Plan Mode에서는 Sub Agent(Plan)가 코드베이스를 분석한 뒤 계획을 제시합니다.  
> 계획을 리뷰하고 수정한 뒤 실행을 승인하면, 잘못된 방향으로 진행될 위험이 줄어듭니다.

## 1.2 지시 작성 체크리스트

작업 지시를 내리기 전 아래를 빠르게 점검합니다.

```
□ 목적과 의도가 명시되어 있는가? (Why)
□ 무엇을 하는지 명확한가? (What)
□ 범위가 정해져 있는가? (Scope) — "~는 제외"가 있는가?
□ 기술 스택/방법을 지정했는가? (How)
□ 참고 자료를 명시했는가? (Reference) — URL, 파일 경로
□ 완료 기준이 있는가? (Verify) — 테스트, 명령어, 기대 결과
□ 한 번에 너무 많은 것을 요구하지 않았는가?
```

## 1.3 결과물을 파일로 저장하고 갱신하라

### AI Agent는 기억을 물리적으로 보존하지 않는다

AI Agent와의 대화에서 결정된 사항(설계 방향, 아키텍처 선택, API 스펙 등)은 **세션의 컨텍스트 윈도우 안에만 존재**합니다. 이것은 메모리(RAM)와 같아서, 세션이 끝나면 사라집니다.

```
[세션 중]
  사용자: "인증은 JWT로 하자"
  AI Agent: "네, JWT로 진행합니다"
  사용자: "DB는 PostgreSQL로"
  AI Agent: "네, PostgreSQL로 설계합니다"
  ... (1시간 동안 10가지 결정)

[세션 종료 — 실수로 터미널을 닫음]
  → 10가지 결정 사항 전부 소실
  → 새 세션에서 처음부터 다시 설명해야 함
```

### 실제로 발생하는 문제

| 상황 | 결과 |
|------|------|
| 실수로 터미널 종료 | 진행된 모든 결정 사항 소실 |
| 컨텍스트 자동 압축 (`/compact`) | 앞부분 결정 사항이 요약 과정에서 누락될 수 있음 |
| 다음 날 이어서 작업 | 새 세션이므로 이전 맥락 없음 |
| 팀원에게 인수인계 | 대화 내역만으로는 결정 경위 파악 어려움 |

### 해결 방법 — 결정 사항을 즉시 파일로 저장

작업 과정에서 결정된 사항은 **그때그때 물리적인 파일로 갱신**합니다.  
파일은 세션이 종료되어도 남아 있으므로, 언제든 이어서 작업할 수 있습니다.

**지시 예시 ①: 계획 수립 시**

```
> 사용자 인증 시스템에 대한 계획을 세울 거야.
> 결정된 사항들을 ./todo/Plan_Draft_Auth_System.md로 갱신해줘.
> 갱신하기 전에 나에게 적용할지 물어보고 승인 받아서 진행해.
```

**지시 예시 ②: 설계 진행 시**

```
> API 스펙을 설계해줘.
> 결정된 내용은 ./docs/API_Spec_v1.md에 반영해줘.
> 변경이 생길 때마다 파일을 업데이트하고, 변경 전에 확인받아줘.
```

**지시 예시 ③: 장기 작업 시**

```
> 이 리팩토링은 여러 세션에 걸쳐 진행될 거야.
> 진행 상황을 ./PROGRESS.md에 기록해줘.
> - 완료된 항목: ✅
> - 진행 중: 🔄
> - 미착수: ⬜
> 매 작업 완료 시 이 파일을 갱신해줘.
```

### 권장 파일 구조

```
my-project/
├── todo/                              ← 계획·의사결정 기록
│   ├── Plan_Draft_Auth_System.md
│   ├── Plan_Draft_DB_Migration.md
│   └── Decision_Log.md               ← 주요 결정과 사유 기록
├── docs/                              ← 산출물 문서
│   ├── API_Spec_v1.md
│   └── Architecture_Overview.md
├── PROGRESS.md                        ← 전체 진행 현황
└── CLAUDE.md                          ← 프로젝트 컨텍스트
```

### 파일 저장 지시의 핵심 3원칙

| 원칙 | 설명 | 지시 예시 |
|------|------|----------|
| **즉시 저장** | 결정 즉시 파일에 반영 | "결정될 때마다 바로 파일에 갱신해줘" |
| **승인 후 적용** | 파일 변경 전 확인 | "갱신 전에 나에게 물어보고 승인 받아줘" |
| **경로 지정** | 어디에 저장할지 명시 | "./todo/Plan_Draft_제목.md로 저장해줘" |

> ⚠️ **핵심**: AI Agent에게 "기억해줘"라고 하는 것보다 **"파일로 저장해줘"** 가 확실합니다.  
> 대화 속 기억은 세션과 함께 사라지지만, 파일은 영구히 남습니다.

## 1.4 작업의 시작과 종료 — 컨텍스트 관리 절차

### 새로운 작업을 시작할 때

하나의 세션 안에서도 작업은 여러 번 바뀝니다.  
"DB 설계 → API 구현 → 테스트 작성"처럼 작업이 전환될 때, **이전 작업의 컨텍스트를 정리하지 않으면** AI Agent는 앞서 읽었던 불필요한 파일 내용과 대화 이력에 묻혀 새 작업의 정확도가 떨어집니다.

동일 세션 내에서 작업을 전환할 때가 이 절차의 **주요 적용 시점**이며,  
세션을 완전히 종료하고 새로 여는 경우에도 동일하게 적용됩니다.

**STEP 1. 작업 개요와 AI Agent의 역할 규정**

새 작업의 첫 지시에서 "무엇을 할 것인지"와 "AI Agent가 어떤 역할인지"를 명확히 선언합니다.

```
> 지금부터 사용자 인증 시스템을 설계할 거야.
> 너는 백엔드 아키텍트 역할이야.
> NestJS + Passport.js 기반으로 설계하고,
> 결정 사항은 ./todo/Plan_Draft_Auth.md에 기록해줘.
```

**STEP 2. 불필요한 컨텍스트 제거 또는 압축**

이전 작업의 컨텍스트가 남아 있으면 새 작업을 방해합니다.

| 상황 | 조치 | 명령 |
|------|------|------|
| 이전 작업과 완전히 다른 새 작업 | 컨텍스트 완전 초기화 | `/clear` |
| 이전 작업과 일부 연관된 새 작업 | 직접 지시하여 선택적 압축 | `/compact 인증 관련 결정만 보존` |
| 중요도 낮은 단순 작업 전환 | 새 작업 개요와 AI Agent 역할을 지시하면 자동 압축이 그에 맞춰 정리 | (STEP 1 지시 후 별도 조치 불필요) |

> ⚠️ Claude의 **자동 압축(`auto-compact`)은 중요도를 모릅니다.**  
> 중요한 작업일수록 `/compact <보존할 내용>`으로 직접 제어하세요.  
> 자동 압축은 중요도가 낮은 작업에서만 신뢰합니다.

**STEP 3. 이전 작업에 이어지는 경우 — 컨텍스트 복원**

앞선 작업의 결과를 이어받아야 할 때, 파일로 저장해 둔 산출물을 기반으로 컨텍스트를 복원합니다.  
이것은 동일 세션 내에서 `/clear` 후 이어 작업할 때나, 새 세션에서 이전 작업을 재개할 때 모두 동일합니다.

```
> 이전 세션에서 인증 시스템 설계를 진행했어.
> 아래 파일을 읽고 현재 상황을 파악해줘:
> - ./todo/Plan_Draft_Auth.md (설계 계획)
> - ./PROGRESS.md (진행 현황)
> - ./docs/API_Spec_v1.md (API 스펙)
>
> 파악한 내용을 요약해주고,
> 이어서 진행할 부분을 알려줘.
```

이때 핵심은 **남길 부분과 제거할 부분을 직접 지시**하는 것입니다.

```
> 이전 작업 파일을 읽었으면:
> - DB 스키마 설계 부분은 확정이니 컨텍스트에서 제거해도 돼
> - JWT 토큰 설계 부분은 아직 미확정이니 유지해줘
> - ./todo/Plan_Draft_Auth.md의 "미결 사항" 섹션을 이번 세션의 작업 목표로 삼아줘
```

### 작업이 종료되었을 때

### 작업이 종료되었을 때 (세션 종료 또는 장시간 중단 시)

세션을 닫거나, 오늘 작업을 마무리하고 내일 이어서 할 때의 부가 절차입니다.  
동일 세션 내 작업 전환이라면 위의 STEP 1~3만으로 충분하지만,  
**세션 자체를 종료할 때는 아래까지 수행해야 다음 세션에서 이어받을 수 있습니다.**

**STEP 1. 결과물을 기록으로 보존**

```
> 오늘 작업 결과를 정리해줘:
> - 완성된 코드는 이미 커밋됨 (확인)
> - 설계 결정 사항을 ./todo/Decision_Log.md에 추가해줘
> - 현재 진행 상황을 ./PROGRESS.md에 반영해줘
```

**STEP 2. 참조 자료 현행화**

AI Agent가 다음 세션에서 읽을 참조 자료를 **현재 상태로 업데이트**합니다.  
이 단계가 없으면, 다음 세션의 AI Agent가 변경 전 자료를 읽고 이전 기준으로 작업합니다.

```
> 오늘 변경된 내용을 기준으로 아래 파일들을 현행화해줘:
> - ./docs/API_Spec_v1.md → 새로 추가된 엔드포인트 반영
> - ./CLAUDE.md → 변경된 아키텍처 정보 반영
> - ./todo/Plan_Draft_Auth.md → 완료된 항목 체크, 미결 사항 업데이트
> 변경 전에 승인받아줘.
```

> ⚠️ 참조 자료가 과거 상태로 남아 있으면, 새 세션의 AI Agent는 **이미 변경된 구조를 모르고** 작업합니다.  
> 이것이 "분명 고쳤는데 또 원래대로 돌아갔다"는 문제의 주요 원인입니다.

**STEP 3. TODO 기록 — 다음 세션 인수인계**

이어서 할 작업을 명시적으로 기록하여, 다음 세션(또는 다른 사람)이 바로 이어받을 수 있게 합니다.

```
> 다음 세션에서 이어서 할 작업을 ./todo/TODO_Next.md에 기록해줘:
> - 남은 작업 목록 (우선순위 순)
> - 각 작업의 선행 조건
> - 참고해야 할 파일 경로
> - 주의사항이나 미결 이슈
```

**TODO 파일 예시**

```markdown
# TODO — 다음 세션 작업 목록

## 우선순위 1: Refresh Token 구현
- 선행: Access Token 발급 완료 (✅)
- 참고: ./docs/API_Spec_v1.md > POST /auth/refresh
- 주의: 토큰 만료 시간은 7일로 확정 (Decision_Log.md #12)

## 우선순위 2: 소셜 로그인 (Google OAuth)
- 선행: Refresh Token 완료 후
- 참고: https://docs.nestjs.com/security/authentication#oauth2
- 미결: 콜백 URL 도메인 미확정 (팀장 확인 필요)

## 주의사항
- main 브랜치 직접 커밋 금지, feature/auth-* 브랜치 사용
- 기존 테스트 186개 모두 통과해야 함
```

```
□ 다음 세션을 위한 TODO가 기록되었는가? (남은 작업, 선행 조건, 미결 이슈)
```

## 1.5 지시 작성 체크리스트 (종합)

작업 지시를 내리기 전 아래를 빠르게 점검합니다.

```
□ 목적과 의도가 명시되어 있는가? (Why) — 왜 이 작업을 하는가?
□ 무엇을 하는지 명확한가? (What)
□ 범위가 정해져 있는가? (Scope) — "~는 제외"가 있는가?
□ 기술 스택/방법을 지정했는가? (How)
□ 참고 자료를 명시했는가? (Reference) — URL, 파일 경로
□ 완료 기준이 있는가? (Verify) — 테스트, 명령어, 기대 결과
□ 한 번에 너무 많은 것을 요구하지 않았는가?
□ 결과물을 파일로 저장하라고 했는가? — 경로, 파일명, 승인 절차
□ 작업 종료 시 참조 자료 현행화와 TODO 기록을 지시했는가?
```
