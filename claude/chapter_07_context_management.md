# 심화 2장. 컨텍스트 관리와 토큰 최적화

> AI Agent의 결과 품질은 능력이 아니라 **컨텍스트의 질**에 의해 결정됩니다.

---

## 2.1 AI Agent는 왜 매번 다른 품질의 결과를 내는가

같은 AI Agent에게 같은 수준의 지시를 내려도, 어떤 때는 훌륭한 결과가 나오고 어떤 때는 엉뚱한 결과가 나옵니다. 이것은 AI Agent의 능력이 들쑥날쑥한 것이 아닙니다.

**원인은 두 가지입니다.**

### ① 기초 탐색을 임의로 실행한다

AI Agent는 지시를 받으면, 실행에 앞서 **기초 탐색**을 수행합니다. 어떤 파일을 읽을지, 어떤 코드를 참고할지, 어떤 구조를 전제로 할지를 스스로 판단합니다.

문제는 이 탐색 과정이 **지시에 명시되지 않은 영역**이라는 점입니다.

```
[같은 지시, 다른 탐색 결과]

> "인증 API를 구현해줘"

[세션 A — 탐색 결과가 좋았던 경우]
  → src/auth/ 폴더를 먼저 읽음
  → 기존 인증 패턴을 발견
  → 기존 패턴과 일관된 코드 생성
  → 결과: 우수

[세션 B — 탐색 결과가 빗나간 경우]
  → src/users/ 폴더를 먼저 읽음
  → 인증과 무관한 코드를 컨텍스트에 로드
  → 잘못된 전제 위에서 코드 생성
  → 결과: 의도와 다름
```

### ② 컨텍스트에 따라 같은 지시를 다르게 해석한다

AI Agent는 동일한 지시라도, 그 시점의 컨텍스트 윈도우에 무엇이 들어 있느냐에 따라 해석이 달라집니다. 이전 대화에서 논의한 내용, 읽어들인 파일, MCP로 조회한 결과 — 이 모든 것이 다음 지시의 해석에 영향을 줍니다.

> **결과의 품질이 일정하지 않은 이유는 AI Agent의 능력 차이가 아니라, 세션마다 달라지는 컨텍스트의 차이입니다.**

## 2.2 컨텍스트가 결과 품질을 결정한다

AI Agent의 모든 판단은 **해당 세션에서 축적된 컨텍스트의 맥락**에 의해 결정됩니다.

```
┌─────────────────────────────────────────────────────────┐
│                 컨텍스트 윈도우 (200K 토큰)                │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  시스템 프롬프트 + 도구 설명     ← 고정 영역             │
│  CLAUDE.md 지침                 ← 세션 시작 시 로드      │
│  MCP 도구 설명                  ← 활성화된 서버만큼      │
│  ─────────────────────────────                          │
│  AI Agent가 읽은 파일들          ← 기초 탐색 결과        │
│  이전 대화 내용                  ← 대화가 쌓일수록 증가  │
│  도구 실행 결과                  ← 명령 실행 결과물      │
│  ─────────────────────────────                          │
│  ▼ 이 전체 맥락을 기반으로 다음 지시를 해석·실행 ▼       │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

여기서 핵심은: 컨텍스트의 **어떤 내용이 얼마나 비중을 차지하느냐**가 AI Agent의 다음 판단 방향을 좌우한다는 점입니다.

| 컨텍스트 상태 | AI Agent의 판단 | 결과 품질 |
|-------------|----------------|----------|
| 관련 파일이 정확히 로드됨 | 근거 기반 판단 | 높음 |
| 관련 없는 파일이 대량 로드됨 | 노이즈에 의한 판단 | 불안정 |
| 대화가 길어져 지침이 밀려남 | 최근 대화 기반 판단 | 점진적 하락 |
| 이전 실패 시도가 쌓여 있음 | 실패 패턴에 매몰 | 악순환 |

## 2.3 AI Agent의 판단 범위를 좁혀라

심화 1장에서 **명확한 지시(7가지 요소)** 를 다뤘습니다. 그것은 "무엇을 할지"를 좁히는 것이었습니다.

여기서 한 단계 더 나아갑니다. **"어떻게 수행할지를 판단하는 부분"의 범위도 좁혀야 합니다.**

AI Agent가 지시를 받으면 내부적으로 두 가지를 판단합니다:

```
[사용자 지시]
> 사용자 인증 API를 구현해줘

[AI Agent의 내부 판단]

  판단 1: "무엇을 할까?" (What)
    → 사용자 지시에서 파악 — 인증 API 구현

  판단 2: "어떻게 수행할까?" (How to execute)
    → 어떤 파일을 먼저 읽을까?
    → 기존 코드 패턴이 있는지 탐색할까?
    → 어떤 라이브러리를 참고할까?
    → 테스트는 어떤 방식으로 작성할까?
    → ← 이 부분이 임의 탐색의 영역
```

심화 1장의 지시 방법은 **판단 1**을 좁히는 것이고, 이 장에서 다루는 것은 **판단 2**를 좁히는 것입니다.

**판단 2를 좁히는 방법:**

| 방법 | 설명 | 예시 |
|------|------|------|
| **역할 명시** | AI Agent가 수행할 역할을 정확히 규정 | "시니어 백엔드 개발자로서 구현해줘" |
| **참조 파일 지정** | 탐색 대상을 직접 지정하여 임의 탐색 차단 | "src/auth/auth.service.ts 패턴을 따라서" |
| **수행 절차 명시** | 실행 순서를 직접 규정 | "1) 스키마 확인 → 2) 서비스 구현 → 3) 테스트" |
| **판단 기준 제공** | 애매한 상황에서의 판단 원칙 제시 | "확신이 없으면 실행 전에 질문해줘" |

## 2.4 사람이 모든 것을 명시할 수는 없다

여기서 현실적인 한계에 부딪힙니다.

**사람이 매번 모든 참조 파일, 모든 수행 절차, 모든 판단 기준을 지시에 포함시킬 수는 없습니다.**

```
[이상적이지만 비현실적인 지시]
> 사용자 인증 API를 구현해줘.
> - src/auth/auth.service.ts의 패턴을 따를 것
> - src/common/decorators/의 커스텀 데코레이터를 활용할 것
> - prisma/schema.prisma의 User 모델을 참조할 것
> - docs/api-convention.md의 응답 형식을 따를 것
> - tests/auth/의 기존 테스트 패턴을 참고할 것
> - ...

→ 프로젝트의 모든 파일 관계를 사람이 기억하고 있어야 가능
→ 프로젝트가 커질수록 불가능
```

매번 이렇게 지시할 수 없다면, **AI Agent가 스스로 올바른 파일을 찾아갈 수 있는 구조**를 만들어야 합니다.

그 구조가 바로 **산출물 인덱스**와 **작업 유형별 참조 가이드**입니다.

---

## 2.5 산출물 인덱스 — 프로젝트의 지도를 만들어라

### 산출물 인덱스란?

프로젝트 내의 주요 산출물 파일의 **구조와 진입점을 인덱스화한 문서**입니다. AI Agent가 "어떤 파일을 먼저 읽어야 하는지"를 판단할 때 지도 역할을 합니다.

사람이 새로운 프로젝트에 투입되었을 때 가장 먼저 찾는 것이 "프로젝트 구조 문서"인 것과 같습니다. AI Agent도 이 지도가 있으면 임의 탐색 대신 **목적 지향적 탐색**을 합니다.

### 인덱스가 없을 때 vs 있을 때

```
[인덱스 없음]
  AI Agent: "인증 API를 만들라고 했으니... 일단 전체 파일을 훑어보자"
  → 100개 파일을 읽음 → 컨텍스트 50% 소모 → 정작 작업할 공간 부족

[인덱스 있음]
  AI Agent: "PROJECT_INDEX.md를 보니 인증은 src/auth/에,
            API 규칙은 docs/api-convention.md에 있군"
  → 3개 파일만 읽음 → 컨텍스트 5% 소모 → 작업 공간 충분
```

### PROJECT_INDEX.md 작성 예시

```markdown
# PROJECT_INDEX — 프로젝트 산출물 인덱스

> 이 파일은 AI Agent가 프로젝트를 탐색할 때 진입점으로 사용합니다.
> 새로운 작업을 시작할 때 이 파일을 먼저 읽으세요.

## 프로젝트 개요
- 주간업무보고 시스템 (NestJS 11 + React 18)
- 약 9명 팀, DX/AX 2개 서브그룹

## 핵심 디렉토리 구조

| 디렉토리 | 역할 | 진입 파일 |
|----------|------|----------|
| src/api/ | REST API 라우트 | src/api/README.md |
| src/auth/ | 인증·인가 | src/auth/auth.module.ts |
| src/services/ | 비즈니스 로직 | src/services/README.md |
| src/components/ | React 컴포넌트 | src/components/index.ts |
| prisma/ | DB 스키마·마이그레이션 | prisma/schema.prisma |
| docs/ | 설계·규칙 문서 | docs/README.md |
| tests/ | 테스트 코드 | tests/README.md |

## 핵심 설계 문서

| 문서 | 용도 | 참고 시점 |
|------|------|----------|
| docs/api-convention.md | API 응답 형식, 에러 코드 규칙 | API 구현·수정 시 |
| docs/auth-design.md | 인증 아키텍처 설계 | 인증 관련 작업 시 |
| docs/db-schema.md | ERD 및 테이블 설명 | DB 스키마 변경 시 |
| docs/git-workflow.md | 브랜치 전략, 커밋 규칙 | 커밋·PR 시 |

## 기술 스택 요약
- Backend: NestJS 11, TypeScript, Prisma, PostgreSQL
- Frontend: React 18, Tailwind CSS
- Test: Jest, Testing Library
- CI: GitHub Actions
```

### 인덱스의 핵심 원칙

| 원칙 | 설명 |
|------|------|
| **진입점을 명시** | 디렉토리가 아니라 "이 파일부터 읽어라"를 지정 |
| **용도별 분류** | 파일이 아닌 "이 작업을 할 때 이것을 참고"로 연결 |
| **가볍게 유지** | 인덱스 자체가 길면 의미 없음. 50~100줄 이내 |
| **현행화** | 파일 구조가 변경되면 인덱스도 갱신 (작업 종료 시 체크) |

## 2.6 작업 유형별 참조 가이드

산출물 인덱스가 "프로젝트의 지도"라면, 작업 유형별 참조 가이드는 **"이 작업을 할 때는 이 지도의 이 부분을 봐라"** 라고 알려주는 안내판입니다.

### 왜 필요한가?

AI Agent는 "인증 API를 구현해줘"라는 지시를 받으면, 인증과 관련된 파일을 찾아야 합니다. 인덱스가 있더라도, 인증 작업에 **어떤 파일 조합이 필요한지**는 여전히 AI Agent가 판단해야 합니다.

작업 유형별 참조 가이드는 이 판단마저 사람이 미리 정의해둔 것입니다.

### CLAUDE.md에 작성하는 방식

```markdown
## 작업 유형별 참조 가이드

### API 신규 구현 시
1. docs/api-convention.md — 응답 형식, 에러 코드 확인
2. src/api/ 내 유사 API 파일 — 기존 패턴 참고
3. prisma/schema.prisma — 관련 모델 확인
4. tests/ 내 유사 테스트 — 테스트 패턴 참고

### API 수정·버그 수정 시
1. 해당 API 파일 + 관련 서비스 파일
2. tests/ 내 해당 API 테스트 — 기존 테스트 확인
3. 수정 후 npm test 실행하여 회귀 확인

### DB 스키마 변경 시
1. docs/db-schema.md — 현재 ERD 확인
2. prisma/schema.prisma — 스키마 수정
3. 마이그레이션 생성 후 docs/db-schema.md 현행화
4. 관련 서비스 파일의 영향 범위 확인

### 프론트엔드 컴포넌트 추가 시
1. src/components/index.ts — 기존 컴포넌트 목록 확인
2. src/components/ 내 유사 컴포넌트 — 패턴 참고
3. Tailwind 클래스 순서: docs/frontend-convention.md

### 인증·인가 관련 작업 시
1. docs/auth-design.md — 인증 아키텍처 확인
2. src/auth/ — 기존 인증 코드 확인
3. 변경 시 기존 인증 테스트 전체 실행

### 배포·인프라 작업 시
1. docs/deploy-guide.md — 배포 절차 확인
2. .github/workflows/ — CI/CD 파이프라인 확인
3. docker-compose.yml — 컨테이너 구성 확인
```

### 참조 가이드의 효과

```
[참조 가이드 없음]
  사용자: "결제 API 구현해줘"
  AI Agent: (어디부터 볼지 몰라서 전체 탐색)
  → 토큰 낭비 + 잘못된 파일 참조 가능성

[참조 가이드 있음]
  사용자: "결제 API 구현해줘"
  AI Agent: (CLAUDE.md의 "API 신규 구현 시" 참조)
  → 1. api-convention.md 읽기
  → 2. 유사 API 패턴 확인
  → 3. 스키마 확인
  → 4. 구현 시작
  → 목적 지향적 탐색, 토큰 절약, 일관된 품질
```

## 2.7 산출물 인덱스와 참조 가이드의 관계

```
┌──────────────────────────────────────────────────────────┐
│  CLAUDE.md                                               │
│  ├── 프로젝트 개요, 기술 스택, 빌드 명령                  │
│  ├── 작업 유형별 참조 가이드  ←── "이 작업엔 이걸 봐라"   │
│  └── 핵심 규칙                                           │
├──────────────────────────────────────────────────────────┤
│  PROJECT_INDEX.md                                        │
│  ├── 디렉토리 구조 + 진입 파일  ←── "프로젝트 전체 지도"   │
│  └── 핵심 설계 문서 목록                                  │
├──────────────────────────────────────────────────────────┤
│  docs/*.md                                               │
│  ├── api-convention.md                                   │
│  ├── auth-design.md        ←── "실제 참고 자료"           │
│  └── db-schema.md                                        │
└──────────────────────────────────────────────────────────┘

AI Agent의 탐색 경로:
  ① CLAUDE.md → "API 구현이니까 참조 가이드를 보자"
  ② 참조 가이드 → "api-convention.md와 유사 API를 보라고 되어 있군"
  ③ PROJECT_INDEX.md → "API 디렉토리 진입점은 src/api/README.md"
  ④ 실제 파일 읽기 → 목적에 맞는 최소한의 파일만 로드
```

이 구조가 갖춰지면:

- **임의 탐색이 줄어듭니다** → 토큰 낭비 방지
- **탐색 경로가 일정합니다** → 결과 품질의 편차 감소
- **사람이 매번 참조 파일을 지정하지 않아도 됩니다** → 지시가 간결해짐

## 2.8 실전 적용 체크리스트

### 프로젝트 셋업 시

```
□ PROJECT_INDEX.md를 작성했는가?
  - 핵심 디렉토리 + 진입 파일이 명시되어 있는가?
  - 핵심 설계 문서 목록이 포함되어 있는가?
□ CLAUDE.md에 작업 유형별 참조 가이드가 포함되어 있는가?
  - 자주 수행하는 작업 유형이 모두 커버되는가?
  - 각 유형별로 참조할 파일과 순서가 명시되어 있는가?
□ CLAUDE.md에 "새 작업 시작 시 PROJECT_INDEX.md를 먼저 읽어라"가 명시되어 있는가?
```

### 작업 지시 시

```
□ 해당 작업 유형에 맞는 참조 가이드가 CLAUDE.md에 있는가?
  - 있다면: 별도 참조 지시 없이 지시만 내리면 됨
  - 없다면: 참조할 파일을 지시에 직접 포함
□ 새로운 유형의 작업이라면, 완료 후 참조 가이드에 추가했는가?
```

### 작업 종료 시

```
□ 파일 구조가 변경되었다면 PROJECT_INDEX.md를 현행화했는가?
□ 새로운 설계 문서가 추가되었다면 인덱스에 반영했는가?
□ 새로운 작업 유형이 발견되었다면 참조 가이드에 추가했는가?
```

## 2.9 컨텍스트 격리 — 개발자에게 테스트를 맡기지 말라

### "자기가 만든 코드를 자기가 테스트하면 안 된다"

소프트웨어 개발에서 오래된 원칙이 있습니다. **개발자에게 테스트를 맡기지 말라.** 자신이 어디를 어떻게 개발했는지 알기 때문에, 테스트할 때 무의식적으로 "이건 당연히 되겠지", "이 경로는 문제없을 거야"라고 넘어갑니다. 실제로 문제가 있는 엣지 케이스를 확인하지 않고, 자신이 구현한 정상 경로만 검증하게 됩니다.

**AI Agent도 이 문제에서 자유롭지 않습니다.**

### 동일 세션에서 구현과 검증을 하면 생기는 문제

AI Agent가 코드를 구현한 세션에는 **구현 과정의 모든 맥락**이 컨텍스트에 남아 있습니다. 어떤 설계 판단을 했는지, 어떤 파일을 수정했는지, 어떤 방향으로 문제를 해결했는지가 모두 기억 속에 있습니다.

이 상태에서 "테스트해줘"라고 하면:

```
[동일 세션에서 구현 + 검증]

  AI Agent의 컨텍스트:
  ├── 구현 과정에서의 설계 판단 (이미 알고 있음)
  ├── 수정한 파일 목록 (이미 알고 있음)
  ├── 해결한 문제의 방향 (이미 알고 있음)
  └── 남아 있는 기술 부채 (합리화할 수 있음)

  검증 시 발생하는 문제:
  · "이 부분은 내가 이렇게 구현했으니 당연히 동작한다" → 스킵
  · "이 에러 처리는 3회 재시도로 설계했으니 맞다" → 설계 자체를 의심 안 함
  · "이 경로는 내가 의도한 대로 흐른다" → 의도하지 않은 경로 미검증
  · 자신이 추론으로 결정한 부분을 결함으로 인식하지 못함
```

이것은 사람 개발자가 자기 코드를 테스트할 때 빠지는 함정과 정확히 같습니다. **아는 것이 오히려 독이 되는 상황**입니다.

### 검증은 반드시 새로운 세션에서

검증을 새로운 세션에서 수행하면, AI Agent는 **구현 과정의 맥락 없이 순수하게 결과물만 보고 판단**합니다.

```
[새로운 세션에서 검증]

  AI Agent의 컨텍스트:
  ├── CLAUDE.md 지침 (깨끗하게 로드)
  ├── 구현결과 리포트 (문서화된 사실만)
  └── 실제 코드 (있는 그대로)

  검증 시 차이점:
  · 구현 의도를 모름 → 코드가 스스로 설명하지 못하면 결함으로 인식
  · 설계 판단을 모름 → 판단의 타당성을 독립적으로 평가
  · 추론으로 채운 부분을 모름 → 명시되지 않은 동작을 의심
  · 대화 흐름의 편향 없음 → 지침 기반 객관적 검증
```

### 검증 지시 방법

구현이 끝나면, **구현결과 리포트**를 파일로 저장하고, 새 세션에서 이 리포트를 참조하여 검증하도록 지시합니다.

**구현 세션에서:**

```
> 구현이 완료되었으면, 아래 내용을 포함한 구현결과 리포트를 
  ./reports/auth-impl-report.md에 저장해줘.
>
> - 구현 범위 (무엇을 했는가)
> - 변경된 파일 목록
> - 주요 설계 판단과 그 이유
> - 실행한 테스트와 결과
> - 알려진 제한사항 또는 미해결 이슈
> - 검증 시 확인해야 할 항목
```

**새 세션에서:**

```
> ./reports/auth-impl-report.md를 읽고, 해당 구현을 검증해줘.
> - 구현 범위가 요구사항(docs/auth-requirements.md)과 일치하는지
> - 코드 품질, 보안 취약점, 성능 이슈
> - 리포트에 명시된 '검증 시 확인 항목'을 모두 수행
> - 발견된 문제를 ./reports/auth-review-report.md에 저장
> - 추가로 확인이 필요한 항목이 있으면 별도 제시
```

### 유용한 팁: 백그라운드 실행으로 세션 자동 분리

Claude Code에서 구현이 끝난 후, 검증을 **백그라운드로 실행하도록 지시**하면 자동으로 새로운 세션에서 실행됩니다. 현재 세션의 컨텍스트와 완전히 격리된 상태에서 검증이 수행됩니다.

```
> 구현이 완료되었어.
> 이제 아래 검증 작업을 백그라운드에서 실행해줘.
>
> 백그라운드 작업:
>   1. ./reports/auth-impl-report.md를 참조하여 구현 결과를 검증
>   2. npm test 실행 결과를 ./reports/auth-test-output.log에 저장
>   3. 검증 리포트를 ./reports/auth-review-report.md에 저장
>   4. 완료되면 결과 파일 경로를 출력
>
> 나는 이 세션에서 다른 작업을 이어서 진행할게.
```

**백그라운드 실행의 장점:**

| 장점 | 설명 |
|------|------|
| 컨텍스트 격리 | 구현 맥락 없이 순수하게 결과물만 검증 |
| 시간 절약 | 검증을 기다리지 않고 다음 작업 진행 가능 |
| 병렬 작업 | 구현(현재 세션) + 검증(백그라운드) 동시 수행 |
| 객관성 확보 | 구현자의 편향이 검증에 영향을 주지 않음 |

### 고급 팁: 별도 프로세스로 실행하고 모니터링

더 명시적인 격리가 필요한 경우, 별도 프로세스로 실행하고 출력을 모니터링하도록 지시할 수 있습니다.

```
> 아래 검증 스크립트를 별도 프로세스로 실행하고,
> 출력 결과를 ./reports/verification-output.log에 기록해줘.
> 프로세스 실행 후 완료될 때까지 로그를 모니터링하고,
> 실패 항목이 있으면 즉시 보고해줘.
>
> 실행할 검증:
>   npm test -- --coverage
>   npm run lint
>   npm run type-check
```

```

```
