# 개요. AI Agent란 무엇인가

> AI Agent 사용자 교육 시리즈 | 작성일: 2026-03-27 | v0.1 (초안)

---

## 0.1 AI Agent를 어떻게 바라볼 것인가?

AI Agent를 처음 접하면 자연스럽게 이런 질문이 떠오릅니다.

> "이건 도구인가, 아니면 뭔가 다른 것인가?"

사람마다 기대가 다릅니다.

| 관점                      | 기대                        | 실제 사용 시 문제                                      |
| ------------------------- | --------------------------- | ------------------------------------------------------ |
| **유용한 도구**           | 버튼 누르면 결과가 나온다   | "알아서 해줘"로 시작 → 의도와 다른 결과 → 실망         |
| **편리한 자동화**         | 반복 작업을 대신 처리해준다 | 단순 반복에는 강하지만, 맥락 없는 자동화는 오류를 양산 |
| **내 일을 대신하는 존재** | 시키면 다 해준다            | 검증 없이 결과를 수용 → 품질 사고 발생                 |

이 관점들이 틀린 것은 아닙니다. 하지만 어느 하나만으로는 AI Agent를 제대로 활용하기 어렵습니다.

---

## 0.2 이 가이드의 관점 — AI Agent는 협업 대상이다

이 가이드는 AI Agent를 **Cowork 대상**, 즉 협업 파트너로 봅니다.

도구는 지시한 대로만 움직입니다. 하지만 AI Agent는 맥락을 읽고, 부족한 정보를 추론하며, 때로는 더 나은 방향을 제안합니다. 동시에, 잘못된 방향으로 확신을 갖고 진행하기도 합니다.

이 특성은 사람 동료와 비슷합니다. 그래서 도구가 아니라 **동료로서 협업하는 방식**이 가장 효과적입니다.

**협업 파트너로서의 AI Agent:**

- 내가 부족한 영역을 보완해주는 동료
- 지시가 명확할수록 좋은 결과를 내는 동료
- 중간중간 확인하고 방향을 잡아줘야 하는 동료
- 결과물을 검증하는 것은 여전히 내 책임

> 💡 "AI Agent가 내 일을 대신한다"가 아니라,  
> **"AI Agent와 함께 일하면 혼자보다 더 많은 일을 더 잘 해낼 수 있다"** 가 올바른 기대입니다.

## 0.3 실전 예시 ① — 시니어가 주니어 역할의 AI Agent와 협업하는 경우

### 상황

팀에 일은 많은데 주니어 인력이 없습니다.  
시니어 개발자가 설계부터 코딩, 문서화, 테스트까지 전부 직접 하고 있습니다.  
설계에 집중해야 하는데, 구현과 문서화에 시간을 빼앗기는 구조입니다.

### AI Agent의 역할: 주니어 개발자

```
[역할 분담]

시니어 (사람)                    AI Agent (주니어 역할)
─────────────                   ─────────────────────
① 요구사항 분석·설계              
② 구현 지시 ──────────────→     ③ 코드 구현
                                ④ 단위 테스트 작성
                                ⑤ 수행결과 보고서 작성
⑥ 코드 리뷰·검증  ←────────── 
⑦ 수정 지시 ──────────────→     ⑧ 수정 반영
⑨ 릴리즈 판단·배포
```

### 실제 흐름

**1단계 — 시니어가 설계를 완료하고 구현을 지시합니다.**

```
> 아래 설계에 따라 사용자 인증 API를 구현해줘.
> - 설계 문서: docs/auth-design.md
> - 스택: NestJS + Passport.js + Prisma
> - 범위: 회원가입, 로그인 (소셜 로그인 제외)
> - 완료 후: 구현 결과를 ./reports/auth-impl-report.md에 정리
```

**2단계 — AI Agent가 구현하고 결과를 보고합니다.**

AI Agent는 코드를 작성하고, 테스트를 돌리고, 수행결과 보고서를 파일로 남깁니다.

**3단계 — 시니어가 보고서와 코드를 검증합니다.**

수행결과 보고서를 기준으로 의도대로 구현되었는지 확인합니다.  
문제가 있으면 수정 지시를 내리고, 문제가 없으면 릴리즈를 결정합니다.

### 이 구조의 핵심

| 원칙                      | 설명                                                            |
| ------------------------- | --------------------------------------------------------------- |
| 설계는 사람이 한다        | AI Agent는 설계를 "실행"하는 역할                               |
| 지시는 명확하게           | 주니어에게 "알아서 해"라고 하면 결과를 예측할 수 없는 것과 동일 |
| 결과는 반드시 검증        | 코드 리뷰 없이 주니어 코드를 배포하지 않는 것과 동일            |
| 중간 산출물을 파일로 보존 | 구두 보고가 아닌 문서 기반 소통                                 |

> 시니어는 설계와 검증이라는 **고부가가치 업무에 집중**하고,  
> 구현·문서화·테스트라는 **실행 업무를 AI Agent에 위임**합니다.  
> 주니어가 없어서 시니어가 야근하던 구조가 해소됩니다.

## 0.4 실전 예시 ② — 주니어가 선임 역할의 AI Agent와 함께 성장하는 경우

### 상황

스타트업에 시니어 개발자가 없습니다.  
주니어 개발자가 개발 총괄을 맡았지만, 아키텍처 설계나 기술 판단에서 확신이 부족합니다.  
물어볼 선임이 없어서 매번 혼자 결정하고, 나중에 뒤엎는 일이 반복됩니다.

### AI Agent의 역할: 선임 개발자 (멘토 + 리뷰어)

```
[역할 분담]

주니어 (사람)                    AI Agent (선임 역할)
─────────────                   ─────────────────────
① 요구사항 초안 작성              
② 요구사항 검토 요청 ─────→     ③ 빠진 항목·모순 지적 + 질문
④ 보완·확정                     
⑤ 설계 검토 요청 ─────────→     ⑥ 설계 리뷰 + 대안 제시
⑦ 방향 결정                     
⑧ 구현 (직접 코딩)               
⑨ 코드 리뷰 요청 ─────────→     ⑩ 리뷰 + 개선점 피드백
⑪ 수정·학습                     
⑫ 배포 판단
```

### 실제 흐름

**1단계 — 주니어가 요구사항 초안을 작성하고 검토를 요청합니다.**

주니어가 아는 범위에서 요구사항을 정리합니다. 완벽하지 않아도 됩니다.  
선임에게 초안을 보여주고 "빠진 거 없는지 봐달라"고 하는 것과 같습니다.

```
> 결제 시스템을 새로 만들어야 해.
> 내가 정리한 요구사항 초안은 docs/payment-requirements.md에 있어.
> 이 초안을 검토해줘.
> - 빠진 항목이나 모순되는 부분을 지적해줘
> - 내가 결정해야 하는 사항이 있으면 질문 형태로 정리해줘
> - 결과는 docs/payment-requirements-review.md에 저장해줘
```

**2단계 — AI Agent가 요구사항의 빈틈을 짚어줍니다.**

AI Agent는 초안을 분석하고, 주니어가 미처 고려하지 못한 부분을 질문합니다.

```
[AI Agent 리뷰 결과 예시 — docs/payment-requirements-review.md]

■ 빠진 항목
  - 결제 실패 시 재시도 정책이 정의되지 않음
  - 부분 환불 시나리오가 없음
  - 동시 결제 요청 시 재고 차감 순서 미정의

■ 결정 필요 사항
  Q1. PG사는 확정되었는가? (토스페이먼츠 / 아임포트 / 직접 연동)
  Q2. 정기 결제(구독)도 이번 범위에 포함하는가?
  Q3. 결제 데이터 보관 기간 기준이 있는가? (PCI-DSS 고려 여부)

■ 모순
  - 3.2절 "실시간 알림"과 4.1절 "배치 처리"가 상충
    → 어떤 이벤트는 실시간, 어떤 이벤트는 배치인지 구분 필요
```

**3단계 — 주니어가 답변하고, 요구사항을 확정합니다.**

AI Agent의 질문에 하나씩 답변하면서, 처음에는 생각하지 못했던 항목들이 채워집니다.

```
> 리뷰 고마워. 답변 정리했어.
> - Q1: 토스페이먼츠로 확정. 이유는 기존에 사용 중이라 계약 있음.
> - Q2: 정기 결제는 이번 범위에서 제외. 2차 스프린트에서 진행.
> - Q3: PCI-DSS는 우리 규모에선 과하니까, 카드번호는 저장하지 않는 방식으로.
> - 동시 결제 재고 차감은 비관적 락(Pessimistic Lock)으로 처리.
> - 실시간/배치 구분은 결제 완료·실패는 실시간, 정산은 일 배치로.
>
> 위 내용을 반영해서 docs/payment-requirements.md를 업데이트해줘.
> 변경 전 내 승인을 받고 적용해줘.
```

> 💡 이 과정이 주니어 성장의 핵심입니다.  
> 선임이 "이건 생각해봤어?"라고 물어봐 주는 것만으로도  
> 주니어는 **다음번에 스스로 고려할 항목이 늘어납니다.**

**4단계 — 확정된 요구사항을 기준으로 설계를 상의합니다.**

```
> 확정된 docs/payment-requirements.md 기준으로 아키텍처를 검토해줘.
> - 실행하지 말고, 설계 방안을 2~3가지 제시하고 각각의 장단점을 정리해줘.
> - 우리 상황(3인 팀, 토스페이먼츠 연동, 월 주문 1만 건 이하)을 고려해줘.
> - 결과는 docs/payment-architecture-review.md에 저장해줘.
```

**5단계 — AI Agent가 설계 대안을 제시합니다.**

AI Agent는 확정된 요구사항을 기반으로 기술적 선택지를 비교하여 문서로 정리합니다.  
각 방안의 장단점, 리스크, 추천 사유를 포함합니다.

**6단계 — 주니어가 읽고, 이해하고, 결정합니다.**

**AI Agent의 추천을 그대로 따르는 것이 아니라, 근거를 이해하고 스스로 결정**하는 것이 중요합니다. 이해가 안 되는 부분은 다시 질문합니다.

```
> 방안 2에서 이벤트 소싱을 추천한 이유를 좀 더 설명해줘.
> 우리 규모(월 주문 1만 건 이하)에서도 이 구조가 적절한지?
```

**7단계 — 주니어가 직접 구현하고, AI Agent에게 리뷰를 요청합니다.**

```
> src/payment/ 디렉토리의 코드를 리뷰해줘.
> - 내가 결정한 설계(docs/payment-architecture-review.md의 방안 1)를 따랐는지
> - 확정된 요구사항(docs/payment-requirements.md)을 빠짐없이 구현했는지
> - 보안 취약점이 없는지
> - 개선할 점이 있으면 이유와 함께 알려줘
> - 리뷰 결과를 docs/payment-code-review.md에 저장해줘
```

**8단계 — 피드백을 반영하며 학습합니다.**

AI Agent의 리뷰를 읽고, 왜 그런 피드백이 나왔는지 이해하고, 수정합니다.  
이 과정이 반복되면서 주니어의 요구사항 분석·설계·구현 역량이 함께 성장합니다.

### 이 구조의 핵심

| 원칙                              | 설명                                                       |
| --------------------------------- | ---------------------------------------------------------- |
| AI Agent가 대신 만들어주지 않는다 | 주니어가 직접 구현, AI Agent는 리뷰·조언                   |
| 결정은 사람이 한다                | AI Agent는 선택지와 근거를 제공, 최종 판단은 주니어 몫     |
| 모르는 건 바로 질문               | 선임에게 물어보듯, AI Agent에게 "왜?"를 질문하면 학습 가속 |
| 과정을 문서로 남긴다              | 설계 리뷰, 코드 리뷰 결과가 축적되면 팀의 지식 자산이 됨   |

> 선임이 없어서 성장이 정체되던 주니어가,  
> **AI Agent를 선임 역할로 활용하여 설계·리뷰 피드백을 받으며 역량을 키웁니다.**  
> "대신 해주는 도구"가 아니라 **"함께 성장하는 파트너"** 로 활용하는 것입니다.

## 0.5 AI Agent 협업 도입 절차

두 예시는 역할 배치가 다르지만 공통점이 있습니다.  
개인이든 조직이든 **부족한 부분을 AI Agent로 채우고**, **상호 피드백으로 검증하는 구조**라는 점입니다.

이것을 실제로 도입하려면 다음 4단계를 밟습니다.

| Step  | 단계                            | 핵심 질문                  | 산출물                      |
| :---: | ------------------------------- | -------------------------- | --------------------------- |
|   1   | 부족한 점을 진단한다            | 우리에게 무엇이 부족한가?  | AI Agent에게 부여할 역할    |
|   2   | 수행 방식을 정의한다            | 그 역할은 어떻게 일하는가? | 지침 문서 (CLAUDE.md 등)    |
|   3   | 소단위 업무에 적용하며 개선한다 | 의도대로 동작하는가?       | 개선된 역할 정의·지침       |
|   4   | 효과를 확인하고 범위를 확대한다 | 더 넓은 업무에도 통하는가? | 확대 적용 기준·팀 공유 지침 |

---

### Step 1. 부족한 점을 진단한다

먼저 개인 또는 조직에서 **어디에 병목이 있는지** 파악합니다.  
이 진단 결과가 AI Agent에게 어떤 역할을 부여할지를 결정합니다.

| 진단 질문                 | 시니어 팀 예시                     | 주니어 스타트업 예시          |
| ------------------------- | ---------------------------------- | ----------------------------- |
| 어떤 업무가 밀려 있는가?  | 구현·문서화·테스트                 | 설계 리뷰·코드 리뷰           |
| 왜 밀리는가?              | 주니어가 없어서 시니어가 전부 수행 | 선임이 없어서 검증 없이 진행  |
| AI Agent에게 맡길 역할은? | **주니어 개발자** (실행 위임)      | **선임 개발자** (리뷰·멘토링) |

> 핵심은 "AI Agent에게 무엇을 시킬까"가 아니라,  
> **"우리에게 무엇이 부족한가"에서 출발**하는 것입니다.

### Step 2. AI Agent의 수행 방식을 정의한다

역할이 정해지면, **해당 역할을 현재 수행하는 사람(또는 수행해야 하는 사람)의 방식을 정리**합니다.

AI Agent는 "주니어처럼 해줘"라고 하면 자기 나름대로 해석합니다.  
실제 그 역할을 수행하는 사람이 **자신의 업무 방식을 구체적으로 문서화**해야 AI Agent가 의도대로 움직입니다.

```
[예시 — 시니어가 "주니어 역할"의 수행 방식을 정의]

■ 코드 구현 시
  - 설계 문서를 먼저 읽고, 설계에 없는 판단은 하지 말 것
  - 구현 완료 후 반드시 수행결과 보고서를 작성할 것
  - 보고서에는: 구현 범위, 변경 파일 목록, 테스트 결과, 미해결 이슈

■ 테스트 시
  - 단위 테스트는 Jest로 작성
  - 커버리지 80% 이상 확보
  - 기존 테스트가 깨지면 즉시 보고

→ 이 내용이 CLAUDE.md 또는 별도 지침 파일로 들어갑니다.
```

```
[예시 — 주니어가 "선임 역할"의 수행 방식을 정의]

■ 요구사항 리뷰 시
  - 빠진 항목, 모순, 결정 필요 사항을 질문 형태로 정리
  - "이렇게 해"가 아니라 "이건 고려했는가?"로 유도

■ 코드 리뷰 시
  - 설계 문서 대비 일치 여부 확인
  - 보안 취약점, 성능 이슈 지적
  - 개선점은 이유와 함께 제시 (학습 목적)

→ 수행 방식이 명확할수록 AI Agent의 피드백 품질이 올라갑니다.
```

### Step 3. 소단위 업무에 적용하며 개선한다

처음부터 모든 업무에 적용하지 않습니다.  
**작고 리스크가 낮은 업무 하나**에 먼저 적용하고, 역할 정의와 수행 방식을 조정합니다.

```
[적용 → 피드백 → 개선 사이클]

    ┌──────────────────────────────────────┐
    │  소단위 업무에 적용                     │
    │  (예: API 하나 구현, 문서 하나 리뷰)    │
    └──────────────┬───────────────────────┘
                   ▼
    ┌──────────────────────────────────────┐
    │  결과 확인 — 상호 피드백               │
    │  · 의도대로 수행되었는가?              │
    │  · 지시가 부족했던 부분은?             │
    │  · AI Agent의 판단이 틀렸던 부분은?    │
    └──────────────┬───────────────────────┘
                   ▼
    ┌──────────────────────────────────────┐
    │  역할 정의 · 수행 방식 개선             │
    │  · 지침 파일 업데이트                  │
    │  · 지시 템플릿 보완                    │
    │  · 검증 기준 구체화                    │
    └──────────────┬───────────────────────┘
                   │
                   └──→ 다음 업무에 다시 적용
```

이 사이클을 2~3회 반복하면, 역할 정의와 지침이 해당 업무에 맞게 안정화됩니다.

### Step 4. 효과를 확인하고 범위를 확대한다

소단위에서 효과가 확인되면, **업무 범위와 적용 대상을 점진적으로 넓힙니다.**

| 단계 | 적용 범위      | 확인 포인트                                              |
| ---- | -------------- | -------------------------------------------------------- |
| 1차  | API 1개 구현   | 지시→수행→검증 사이클이 돌아가는가?                      |
| 2차  | 모듈 단위 개발 | 여러 파일에 걸친 작업에서 일관성이 유지되는가?           |
| 3차  | 기능 단위 개발 | 설계→구현→테스트→문서화 전체 흐름이 안정적인가?          |
| 4차  | 팀 전체 적용   | 다른 팀원도 동일한 지침으로 AI Agent를 활용할 수 있는가? |

> 급하다고 처음부터 큰 업무에 적용하면,  
> 역할 정의도 안 된 상태에서 AI Agent가 잘못된 방향으로 대량의 코드를 생성합니다.  
> **작게 시작하고, 검증하고, 넓히는 것**이 가장 빠른 길입니다.
